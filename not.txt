1. Principes et objectifs (à garder tout au long du projet)

Respecter l’identité et les valeurs de l’ONG (clarté, empathie, transparence, confiance).

Prioriser l’utilisateur : facilité pour donateurs, bénéficiaires, partenaires, bénévoles.

Sécurité et confidentialité des données personnelles.

Accessibilité (WCAG AA) et responsive mobile-first.

Performance (pages rapides, images optimisées, CDN).

Facilité de maintenance par des non-techniques (rédacteur/administrateur).

2. Phases de la refonte (ordre recommandé — sans estimation temporelle)

Chaque phase contient livrables, tâches et critères d’acceptation.

Phase A — Discovery & Audit (préparation)

Livrables : rapport d’audit (contenu, technique, sécurité), personas, carte des parties prenantes, objectifs fonctionnels prioritaires.
Tâches :

Réunion(s) avec stakeholders (mission, audience, KPIs, limites budgétaires).

Audit du site existant : contenu (pages, images, médias), analytics, formulaires, SEO de base.

Audit technique : hébergement, CMS, versions logicielles, certificats SSL, erreur 403 (vérifier pourquoi), sauvegardes existantes.

Audit sécurité superficiel (headers visibles, robots.txt, sitemap) et posture RGPD/privacy baseline.

Recensement des intégrations nécessaires : paiements (dons), newsletter, CRM, systèmes partenaires, maps, téléphones locaux.
Critères d’acceptation : document signé listant fonctionnalités minimales (MVP), inventaire contenu, liste d’accès (hébergement, CMS, analytics).

Phase B — Architecture de l’information & contenu

Livrables : sitemap final, modèle de contenu (content types), plan de migration des contenus.
Tâches :

Définir pages principales (ex. Accueil, À propos, Programmes, Actualités, Ressources / Publications, Faire un don, Contact, Partenaires, Rapports annuels, Espace bénévole).

Définir modèles de contenu (article, événement, projet, publication PDF, rapport).

Plan migration : mapping des URLs existantes → nouvelles URLs (préserver SEO, créer redirections 301).
Critères : sitemap validé, mapping URLs et redirections préparées.

Phase C — UX / Wireframes & prototypes

Livrables : wireframes basse fidélité (mobile + desktop), prototypes cliquables (flux clés : don, contact, lire article).
Tâches :

Conception centrée utilisateur : parcours donateur, parcours visiteur cherchant aide, parcours partenaire.

Prototypage des composants critiques : header/footer, menu, formulaire don, formulaire contact, pages d’articles, page projet.

Tests utilisateurs rapides (3–5 personnes) pour valider flows critiques.
Critères : prototypes validés et retours synthétisés.

Phase D — UI Design & Design System

Livrables : UI Kit / design system (couleurs, typographie, composants, variables CSS), maquettes haute-fidélité (desktop/mobile).
Tâches :

Intégrer valeurs de l’ONG dans la charte graphique (ton, images, iconographie, microcopy empathique).

Définir composants réutilisables (cartes projets, boutons CTA, formulaires, modal, badges confiance).

Préparer assets optimisés (SVG pour icônes, images responsive, placeholders).

Préparer instructions d’accessibilité (contrastes, labels, focus states).
Critères : UI Kit livré + maquettes validées.
Note : tu fourniras ton template — j’intègre ton template dans ce design system.

Phase E — Frontend & Intégration (dev)

Livrables : code frontend, composants réutilisables, build optimisé, storybook (optionnel).
Tâches :

Choix technique (voir section « Reco stack » ci-dessous).

Intégration responsive du design system + composants.

Implementation des templates de contenu (articles, pages projets, formulaires).

SEO technique (balises meta, Open Graph, sitemap.xml, robots, structured data pour articles/organisation).

Lazy-load images, responsive srcset, critical CSS, code-splitting.
Critères : pages passent audit Lighthouse (desktop & mobile) avec scores ciblés, validation responsive sur devices clés.

Phase F — Backend, CMS & Intégrations

Livrables : CMS configuré, API si headless, intégrations opérationnelles (paiement, email, analytics).
Tâches :

Configurer CMS (ex. WordPress avec Gutenberg + ACF, ou Headless CMS: Strapi/NetlifyCMS selon besoins).

Système de gestion des utilisateurs / rôles (admin, éditeur, contributeur).

Intégration paiements pour dons (Stripe/PayPal + intégration mobile money locale si nécessaire).

Configurer envoi d’emails transactionnels & newsletter (Sendinblue, Mailchimp, ou service SMTP).

Mise en place sauvegardes automatiques et versionning contenu.
Critères : CMS utilisable par éditeurs non-techniques, paiement test en sandbox validé, emails envoyés correctement.

Phase G — Sécurité & durcissement

Livrables : checklist sécurité, mise en place WAF, scans & rapport.
Tâches :

Forcer HTTPS, vérifier certificat TLS et config (HSTS).

Ajouter en-têtes de sécurité (CSP, X-Frame-Options, X-Content-Type-Options, Referrer-Policy).

Mettre en place WAF ou règles basiques (ex. Cloudflare).

Activer 2FA pour comptes admin, restreindre accès admin (IP allowlist si possible).

Vérification dépendances / correctifs (SCA), config file permissions, backups hors site.

Faire un scan dynamique (OWASP ZAP/Nikto) et un scan CMS (WPScan si WP).
Critères : résultats de scans corrigés ou plan d’action priorisé.

Phase H — Tests & QA

Livrables : rapports QA (fonctionnel, régresion, accessibilité, performance), plan de correction.
Tâches :

Tests fonctionnels (CTAs, formulaires, workflows de don).

Tests cross-browser and cross-device.

Tests d’accessibilité (axe/pa11y) et corrections.

Tests de montée en charge légers (smoke test).
Critères : pass pour checklist de release.

Phase I — Déploiement & Launch

Livrables : procédure de déploiement, rollback plan, DNS et HTTPS validés.
Tâches :

Pipeline CI/CD configuré (build → staging → production).

Switch DNS, vérifier certificats, activer CDN, monitor uptime.

Mettre en place analytics (Google Analytics/Matomo) et suivi conversions (don).
Critères : site en prod, monitoring en place, redirections 301 actives, tests smoke passés.

Phase J — Maintenance & gouvernance post-lancement

Livrables : plan de maintenance, manuel admin, calendrier de backups, SLA de sécurité.
Tâches :

Plan de mise à jour régulière du CMS et plugins.

Monitoring (uptime, erreurs 5xx, logs), alertes de sécurité.

Rapports périodiques (trafic, conversions, performances).

Formation des éditeurs (session + documentation).
Critères : admin formé, doc remise, plan d’action pour incidents.

3. Recommandations techniques (stack & hébergement)

Je propose d’exposer deux options — choisis selon compétence d’administration et budget :

Option A — WordPress classique (recommandé si équipe non-technique)

Pourquoi : interface familière, gestion simple des contenus, large écosystème (plugins pour dons, formulaires, SEO).

Bonnes pratiques : thème enfant, ACF/Gutenberg pour blocs, limiter plugins, utiliser plugin de sécurité + backups, hébergement managé (ex. Kinsta/Cloudways/OVH-managed).

Idéal si tu veux que l’équipe éditoriale puisse modifier sans dev.

Option B — Site statique / headless (Next.js / Gatsby + headless CMS)

Pourquoi : performances très élevées, sécurité renforcée (pas de PHP exposé), meilleure scalabilité.

Inconvénients : coût de dev initial plus élevé, procédures de build/deploy, apprentissage pour l’édition.

Bon pour trafics élevés et besoin strict de performance.

Hébergement / CDN : Netlify / Vercel (headless/static) ou DigitalOcean / AWS Lightsail pour budget contrôle, front + Cloudflare CDN & WAF.
Base de données & backups : sauvegardes quotidiennes, snapshots hors site.

5. Sécurité : checklist actionnable (priorité haute)

Résoudre l’erreur 403 et comprendre si c’est WAF / htaccess / permissions.

Forcer HTTPS + config TLS correcte + HSTS.

Mettre en place en-têtes de sécurité (CSP, X-Frame-Options, etc.).

WAF / rate limiting / protection brute force.

2FA pour comptes admin + restereindre accès WP-Admin.

Scanner régulièrement (SCA, dépendances) et appliquer patchs.

Backups automatisés hors site + test de restauration.

Chiffrement des données sensibles en DB au repos si nécessaire.

Politique de mots de passe, least privilege, journalisation.

6. Performance : checklist actionnable

Images WebP, responsive srcset, lazy-loading.

Utiliser CDN (Cloudflare/fastly).

Gzip/Brotli compression, minification CSS/JS.

Critical CSS inlining, defer non-critical JS.

Cache HTTP + cache purging stratégie.

Pré-render / server-side rendering pour pages publiques.

Audit Lighthouse (mobile + desktop) et correction des “opportunities” / “diagnostics”.

7. Accessibilité & conformité

Cible WCAG 2.1 AA.

Labels explicites, alt pour images, focus order, keyboard navigation.

Tester avec axe, NVDA / VoiceOver.

Politique de confidentialité + consentement cookies (cookie banner) si nécessaire.

8. SEO & analytics

Balises meta optimisées + structured data (Organization, BreadcrumbList, Article).

Robots.txt + sitemap.xml.

Setup analytics (GA4/Matomo), et tracking conversions (dons).

Outils Search Console / Bing Webmaster.
